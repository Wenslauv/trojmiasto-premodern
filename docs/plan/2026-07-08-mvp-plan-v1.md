# План MVP — 2026-07-08 (v1)

Источник: уточненный план сессии.
Статус: Draft (готов к handoff в реализацию).

## Plan: MVP сайт турниров Premodern (уточненный)

Собрать MVP сайта с вкладками Игроки, Ивенты, Колоды (колоды как placeholder) и детальными страницами игрока/ивента. Стек выбирается по критерию скорости и простоты поддержки, а не фиксируется заранее. Источник правды по контенту: новые ивенты; агрегированные данные (players/event summaries) вычисляются в момент добавления ивента и коммитятся вместе с ним. Перед публикацией в GitHub Pages обязателен локальный прогон и тестирование.

## Steps
1. Зафиксировать функциональные границы MVP и контракты данных.
Зависит: нет.
Включает:
- обязательные страницы: /events, /players, /event/:id, /player/:id;
- /decks как вкладка-заглушка без полноценной логики;
- правила клика по строкам и ячейкам из ТЗ.

2. Выбрать стек реализации (нефиксированный).
Зависит: шаг 1.
Критерии:
- корректная работа на GitHub Pages;
- скорость разработки;
- простота локального запуска.
Кандидаты:
- Vite + React + HashRouter;
- SvelteKit static export;
- Vue + Vite.
Рекомендация по рискам/срокам: Vite + React + HashRouter как наименее рискованный для Pages.

3. Спроектировать модель контента как event-first.
Зависит: шаг 1.
Результат:
- исходные данные: только ивенты;
- производные данные: players list, player details, recent-events summary, derived metrics;
- производные данные пересчитываются при добавлении нового ивента и коммитятся в репозиторий вместе с ним.

4. Реализовать pipeline добавления ивента (локально).
Зависит: шаг 3.
Параллельно: шаг 5.
Включает:
- входной формат нового ивента;
- скрипт генерации/пересчета производных json;
- проверку целостности (дубликаты id, формат дат YYYY-MM-DD, валидность X-Y-Z).

5. Реализовать UI страниц MVP.
Зависит: шаг 2.
Параллельно: шаг 4.
Включает:
- таблицу ивентов с кликами по правилам ТЗ;
- таблицу игроков;
- страницу игрока (общий винрейт, любимая колода, история);
- страницу ивента (метаданные, standings, матчи/игры, раунды);
- адаптивность таблиц и горизонтальный скролл для динамического числа раундов.

6. Явно выбрать вариант управления контентом (A/B/C) и зафиксировать в документации.
Зависит: шаг 4.
Option A:
- ручное редактирование json (быстро, но выше риск ошибок).
Option B:
- редактирование source-файла (csv/sheet/json-form) + скрипт пересчета производных данных + коммит исходника и результатов.
Option C:
- админ-форма (не в MVP).
Рекомендация для текущего ТЗ и твоего сценария: Option B в event-first формате.

7. Добавить локальный тестовый контур до Pages.
Зависит: шаги 4-5.
Включает:
- команды: install, generate-data, dev, test/build;
- smoke-чеклист ручной проверки страниц и кликов;
- локальную проверку маршрутов и длинных таблиц;
- проверку, что после добавления нового ивента пересчитанные данные отображаются корректно.

8. Подключить CI и деплой в GitHub Pages после успешного локального прохождения.
Зависит: шаг 7.
Включает:
- workflow build/deploy;
- base path;
- публикацию на Pages;
- post-deploy smoke.

## Verification
1. Локально открыть сайт и проверить все маршруты MVP.
2. Проверить клики на странице ивентов:
строка (кроме ячеек игрок/колода) -> ивент; игрок -> игрок; колода -> placeholder.
3. Проверить страницу игроков:
пустой preferred colors, корректный events count, корректный match win %.
4. Проверить страницу игрока:
общий винрейт, любимая колода, таблица ивентов, формат X-Y-Z.
5. Проверить страницу ивента:
дата, число игроков, место, standings, матчи/игры, динамические раунды.
6. Проверить добавление нового ивента:
после запуска генерации обновляются производные json, изменения коммитятся одним набором.
7. Проверить дату в формате YYYY-MM-DD во всех новых/обновленных данных.
8. После локального pass запустить сборку и деплой на Pages.

## Decisions
- Стек не фиксируем заранее; выбираем по критерию минимального риска деплоя и удобства разработки.
- Event-first данные: исходно храним ивенты, агрегаты считаем при добавлении ивента.
- Производные данные допустимо хранить в репозитории (коммит вместе с новым ивентом).
- До публикации обязателен локальный этап тестирования.
- Страницы колод не входят в эту итерацию, только placeholder-навигация.
